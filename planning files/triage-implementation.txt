# REQUIREMENTS DOCUMENT: Adversarial Triage Loop (Category-Based)

## 1. OVERVIEW

### System Goal
Reduce false positives and assign accurate confidence scores to security findings by running each vulnerability category (A05 Injection, A02 Configuration, A10 Error Handling) through a single-round adversarial debate loop: Prosecutor → Defender → Judge.

### Key Constraint
- **One triage loop per category**, not per finding
- Group all A05 findings together, all A02 together, all A10 together
- Each category gets one final confidence score (1-100) representing how sure we are it's a real risk
- Total: 3 debate loops maximum (one per category)

### NEW SCORING SYSTEM: 1-100 Confidence
- **All three agents (Prosecutor, Defender, Judge) output a single confidence_score 1-100**
- **Higher score = more confident it's a real risk**
- 85-100: Definitely a real vulnerability
- 70-84: Very likely a real vulnerability
- 50-69: Uncertain / mixed evidence
- 30-49: Likely NOT a real vulnerability / possible false positive
- 1-29: Almost certainly a false positive

### Data Flow
```
Raw Git Diff
    ↓
Diff Parser (extract hunks)
    ↓
Run All Detectors (A05, A02, A10)
    ↓
Group findings by category
    ↓
For each category with findings:
    Prosecutor → confidence_score 1-100 (how sure it IS real)
    Defender  → confidence_score 1-100 (can disagree with Prosecutor)
    Judge     → risk_label + reasoning (critical_risk | medium_risk | low_risk | false_positive)
    ↓
Output: Up to 3 CategoryTriageVerdicts (one per category with risk labels for user review)
    ↓
Reporter: Format into markdown/JSON report for user evaluation
```

## 2. DATA MODEL REQUIREMENTS

### 2.1 CategoryTriageVerdict
**Purpose:** Represents the final verdict for all findings in one attack category (A05/A02/A10)

**What it must contain:**
- Category identifier (A05, A02, or A10)
- List of all VulnerabilityFinding objects in this category
- List of all DiffHunk objects relevant to these findings
- Prosecutor's argument (text explaining why findings are real)
- **Prosecutor's confidence score (integer 1-100)** where 85-100=definitely real, 50-69=uncertain, 1-29=likely FP
- Defender's argument (text explaining why findings may be false positives)
- **Defender's confidence score (integer 1-100)** same scale
- Judge's reasoning (text explaining the risk assessment and recommendation)
- **Judge's risk label (string)** → one of: "critical_risk", "medium_risk", "low_risk", "false_positive"
- Metadata: total count of findings in category
- **Risk assessment details:**
  - risk_label: Judge's final verdict label (not auto-derived, deliberate decision)
  - severity_score: derived from risk_label mapping (critical→8-10, medium→5-7, low→2-4, FP→1)

**Constraints:**
- Prosecutor & Defender confidence scores must be int 1-100 (required fields)
- Judge's risk_label must be one of 4 options (required field, no auto-derivation)
- All reasoning texts must be 150-400 words (Prosecutor, Defender, Judge)
- Judge's reasoning must explain the risk assessment and provide actionable guidance for user decision-making

---

### 2.2 SecurityReport
**Purpose:** Final aggregated report containing verdicts for all categories

**What it must contain:**
- Timestamp of report generation
- List of CategoryTriageVerdict objects (0-3 verdicts, only for categories with findings)
- Summary text (human-readable overview)

**Constraints:**
- Must support empty report (if no findings detected by any detector)
- Verdicts should be ordered by severity (descending)

- Keep ParsedDiff unchanged
- These are inputs to the triage loop, not outputs

## 3. PROSECUTOR AGENT REQUIREMENTS

### 3.1 Purpose
Argue that all findings in a given attack category ARE real, exploitable vulnerabilities. Red team / attack perspective. Output a confidence score 1-100 on how sure they are.

### 3.2 Input Requirements
- Category name (A05, A02, or A10)
- List of VulnerabilityFinding objects for this category
- List of DiffHunk objects that contain the vulnerable code

### 3.3 Output Requirements (NEW SCORING SYSTEM)
- **confidence_score: integer 1-100**
  - 85-100: Definitely exploitable (obvious attack path in diff)
  - 70-84: Very likely exploitable (strong evidence)
  - 50-69: Moderate confidence (real but possible mitigations)
  - 30-49: Low confidence (could be false positive)
  - 1-29: Very low confidence / likely false positive
- **reasoning: text 150-400 words (explaining the confidence score)**
- JSON response that can be parsed without errors

### 3.4 Functional Requirements
- Must analyze **patterns** across multiple findings in category
- Must reference specific code snippets from hunks when making argument
- Must use attack perspective: how would attacker exploit this?
- Must explain methodology for arriving at confidence score (e.g., "I scored 82 because: source is clearly user input (HIGH confidence), data flows to SQL query (HIGH), no parameterization (HIGH), reachable from public endpoint (HIGH)")
- Temperature must be deterministic (0.1) - same diff should produce same reasoning

### 3.5 Category-Specific Guidance
For **A05 (Injection):**
- Assess: untrusted input sources, dangerous sinks (SQL, subprocess, eval), missing parameterization
- High confidence if: f-strings/format in query + user input clearly visible + no params
- Lower confidence if: query is read-only, input comes from internal config, or validation visible

For **A02 (Configuration):**
- Assess: CORS settings, exposed secrets, debug flags, missing headers
- High confidence if: CORS='*' on whole API + public endpoint + could leak data
- Lower confidence if: CORS only on staging, keys have limited permissions, debug is internal only

For **A10 (Error Handling):**
- Assess: sensitive data in errors, uncaught exceptions, null deref, fail-open
- High confidence if: stack trace exposed to user endpoint + contains DB paths, or auth failure returns success
- Lower confidence if: error logged only (not shown to user), exception caught at higher level

## 4. DEFENDER AGENT REQUIREMENTS

### 4.1 Purpose
Counter the Prosecutor's argument. Explain why findings may be false positives or lower-risk than claimed. Devil's advocate perspective. Output a confidence score 1-100 on whether it's a real risk (can disagree with Prosecutor).

### 4.2 Input Requirements
- Category name
- List of VulnerabilityFinding objects
- List of DiffHunk objects
- Prosecutor's argument (full text - what Defender is countering)
- **Prosecutor's confidence score (1-100)** - Defender knows Prosecutor's confidence

### 4.3 Output Requirements (NEW SCORING SYSTEM)
- **confidence_score: integer 1-100**
  - 85-100: Yes, it IS a real risk (agrees with Prosecutor)
  - 70-84: Likely IS a real risk (mostly agrees)
  - 50-69: Uncertain / mixed signals (could go either way)
  - 30-49: Likely NOT a real risk (disagrees with Prosecutor)
  - 1-29: Definitely NOT a real risk / false positive (strong disagreement)
- **reasoning: text 150-400 words**
- JSON response that can be parsed without errors

### 4.4 Functional Requirements (NEW)
- Must directly address Prosecutor's claims (reference specific parts of their argument)
- Can agree with Prosecutor and give HIGH confidence (50-100)
- Can disagree and give LOW confidence (1-49)
- Must explain why confidence differs from Prosecutor's score
- Temperature should be deterministic (0.1) - consistent reasoning
- If Prosecutor scored 85, Defender might score 35 (strong disagreement) or 78 (minor tweaks)

### 4.5 Defense Strategies (Guidelines for Defenders)
Question source trustworthiness: 
- "Input comes from internal config, not user"
- "Environment variable with restricted access"
- "This is admin-only setting"

Challenge sink dangerousness: 
- "Query is read-only, not admin"
- "SQL execution is sandboxed"
- "This is logging-only, not execution"

Identify hidden protections: 
- "There's validation in the calling function"
- "Authentication check happens before this code"
- "Input is sanitized by middleware"

Highlight limitations: 
- "This code path requires special permissions"
- "Actual impact is low even if exploited"
- "This is test/staging code, not production"

## 5. JUDGE AGENT REQUIREMENTS

### 5.1 Purpose
Synthesize Prosecutor and Defender arguments into a practical risk assessment for user review. Neutral, evidence-based decision-maker. Assigns risk label (not a confidence score) allowing users to evaluate and decide if action is needed.

### 5.2 Input Requirements
- Category name
- List of VulnerabilityFinding objects
- List of DiffHunk objects
- Prosecutor's argument + **Prosecutor's confidence score**
- Defender's argument + **Defender's confidence score**

### 5.3 Output Requirements (RISK LABELS FOR USER REVIEW)
- **risk_label: string** (one of: "critical_risk", "medium_risk", "low_risk", "false_positive")
  - **critical_risk**: Clear evidence of exploitable vulnerability. User should prioritize fixing.
  - **medium_risk**: Finding has merit but may have context/mitigations. User should review and evaluate.
  - **low_risk**: Finding is minor or context suggests low impact. User may defer or document reasoning.
  - **false_positive**: Unlikely to be real risk. User should close/ignore.
- **reasoning: text 200-300 words (explaining why this risk label, actionable guidance for user)**
- JSON response parsed as CategoryTriageVerdict

### 5.4 Functional Requirements
- Must weigh both Prosecutor and Defender arguments fairly (not favor Prosecutor by default)
- Must classify into one of 4 risk labels (not a spectrum—clear decision for user action)
- Must explain reasoning for the risk label choice
- Must explain which arguments were stronger and why
- Must provide actionable guidance: "User should fix this", "User should review context", "User can defer", etc.
- Temperature must be 0.0 (deterministic) - same inputs always produce same verdict
- Output should help user make confident decision on whether to act

### 5.5 Decision Framework (RISK-BASED, NOT SCORING)
Judge should classify by considering:

**Evidence Weight:**
- Does code evidence strongly support Prosecutor? → Lean critical/medium
- Does Defender identify valid, visible mitigations? → Lean medium/low
- Is vulnerability easily exploitable in real scenario? → Lean critical

**Argument Quality:**
- Is Prosecutor citing specific code patterns? (high quality)
- Is Defender identifying real constraints/validations? (high quality)
- Is either argument speculative or generic? (lower confidence in label)

**Risk Magnitude:**
- If exploited, what's the impact? (data loss, auth bypass, etc.)
- How accessible is the attack path? (public endpoint vs. admin-only)
- Are there layers of defense even if one fails?

**Final Mapping (Decision Tree):**
```
Prosecutor 85+ AND Defender <40
  → CRITICAL_RISK (user should fix immediately)

Prosecutor 70-84 OR (Prosecutor 60+ AND Defender <50)
  → MEDIUM_RISK (user should review and evaluate context)

Prosecutor <50 AND Defender >60
  OR Prosecutor 40-60 AND Defender 50-70
  → LOW_RISK (user may defer or document)

Prosecutor <30 OR (Defender 80+ AND clear reasoning)
  → FALSE_POSITIVE (user can close)
```

## 6. ORCHESTRATOR REQUIREMENTS (UPDATED FOR NEW SCORING)

### 6.1 Purpose
Chain all components: parser → detectors → (group by category) → prosecutor/defender/judge → reporter

### 6.2 Input Requirements
- Raw git diff string (from git diff command or git hook)

### 6.3 Output Requirements
- SecurityReport object (containing 0-3 CategoryTriageVerdicts, each with confidence scores)

### 6.4 Workflow Requirements

**Step 1: Parse Diff**
- Use existing diff_parser.py
- Extract hunks from raw diff
- Output: List[DiffHunk]

**Step 2: Run All Detectors**
- In parallel: detect_injection(), detect_configuration(), detect_error_handling()
- Output: 3 separate List[VulnerabilityFinding]

**Step 3: Group by Category**
- A05 findings → group all A05 together
- A02 findings → group all A02 together
- A10 findings → group all A10 together
- Filter out empty categories (skip if no findings)

**Step 4: Triage Each Category (Sequential)**
- For each non-empty category:
  - Call prosecutor.argue_category() → get confidence_score 1-100 + reasoning
  - Call defender.counter_category() → get confidence_score 1-100 + reasoning, can disagree with Prosecutor
  - Call judge.decide_category() → get risk_label (critical_risk/medium_risk/low_risk/false_positive) + reasoning for user
  - Append result to verdicts list
  
**Step 5: Error Handling**
- If Prosecutor fails: Use fallback (confidence=50, reasoning="Analysis failed")
- If Defender fails: Use default (confidence=50, reasoning="Defender analysis skipped")
- If Judge fails: Use fallback (confidence=50, reasoning="Judge analysis failed, manual review recommended")
- Log all errors with context

**Step 6: Generate Report**
- Call reporter.generate_report(verdicts)
- Sort by final_confidence descending
- Output: SecurityReport object with all verdict data

### 6.5 Functional Requirements
- Must handle case where zero, one, two, or all three categories have findings
- Must run prosecutor → defender → judge **sequentially** (each sees previous output)
- Must track and log confidence scores at each stage
- Must validate all outputs match expected Pydantic models before returning
- Must be testable with mock findings and deterministic (T=0.0/0.1)

## 7. REPORTER REQUIREMENTS

### 7.1 Purpose
Format triage verdicts (with confidence scores) into human-readable and machine-readable outputs

### 7.2 Input Requirements
- SecurityReport object (or List[CategoryTriageVerdict])

### 7.3 Markdown Output Requirements
- Header with report timestamp
- Summary section:
  - "Found X critical issues (score 75-100)"
  - "Found Y medium issues (score 50-74)"
  - "Found Z low issues (score 20-49)"
  - "Found W false positives (score 1-19)"
- For each verdict ordered by risk severity (critical → medium → low → false_positive):
  - Category name (A05, A02, A10)
  - **Judge's risk label (CRITICAL_RISK / MEDIUM_RISK / LOW_RISK / FALSE_POSITIVE) - main actionable output**
  - Prosecutor's confidence (e.g., "85/100") with reasoning summary (first 150 words)
  - Defender's confidence (e.g., "25/100") with reasoning summary
  - Judge's reasoning explaining the risk label and recommended user action
  - List of affected files
  - List of vulnerability descriptions
- Footer: "This is an AI-assisted review. Risk labels are assigned by Judge based on Prosecutor/Defender debate. User should review findings and decide on action. Critical issues require immediate attention; Medium issues warrant review; Low issues can be deferred."

### 7.4 JSON Output Requirements
- Machine-readable export of all data
- Include all fields from CategoryTriageVerdict:
  - category, findings, prosecutor_confidence, defender_confidence, final_confidence
  - All reasoning texts
  - Derived verdict_category and severity_score
- Pretty-printed for readability

### 7.5 Functional Requirements
- Must handle 0 verdicts (empty report)
- Must sort verdicts by final_confidence descending
- Must be idempotent (same input produces same output)
- Must not hallucinate or add information not in verdicts
- Should be readable by both humans and machines

## 8. TESTING REQUIREMENTS (UPDATED FOR 1-100 SCORING)

### 8.1 Unit Tests (Per Agent)
Each agent should be testable in isolation with scoring validation:

**Prosecutor Tests:**
- Test A05 with obvious SQL injection → expect confidence 85+
- Test A05 with parameterized safe query → expect confidence 1-30
- Test A02 with CORS='*' on public API → expect confidence 80+
- Test A10 with uncaught exception → expect confidence 75+
- Verify confidence score correlates with code evidence

**Defender Tests:**
- Test countering weak Prosecutor claim (score 40) → Defender should score higher (60+)
- Test accepting strong Prosecutor claim (score 85) → Defender can still score 70+ (partial agreement)
- Test defending false positive → Defender scores 20 or lower
- Verify counter-argument specifically references Prosecutor's reasoning

**Judge Tests:**
- Test Prosecutor 85 + Defender 20 → Judge should score 70+ (follow Prosecutor)
- Test Prosecutor 65 + Defender 70 → Judge should score 65-70 (close call, note it)
- Test Prosecutor 50 + Defender 50 → Judge should score 50 (uncertain, no clear winner)
- Test tie-breaking: when both arguments strong but conflict

### 8.2 Integration Tests (Full Pipeline)
- Test with mock diff containing: 1 SQL injection + 1 unsafe config + 1 uncaught exception
  - Expected: 3 verdicts (A05, A02, A10) with scores
  - Verify each verdict has prosecutor_confidence, defender_confidence, final_confidence
- Test with mock diff containing: only safe code
  - Expected: 0 verdicts OR all verdicts with low confidence (<20)
- Test with mock diff containing: ambiguous findings
  - Expected: verdicts with final_confidence in 40-60 range

### 8.3 Scoring Consistency Tests
- Same SQL injection example → Prosecutor consistently scores 85+ (deterministic with T=0.1)
- Same benign config change → Prosecutor consistently scores <30
- Same findings + Prosecutor 85 → Judge should consistently derive verdict "critical_risk"

### 8.4 Output Validation Tests
- Verify JSON outputs can be parsed into Pydantic models
- Verify Prosecutor/Defender confidence_score is int 1-100 (not float, not out of range)
- Verify Judge's risk_label is one of 4 valid options: critical_risk, medium_risk, low_risk, false_positive
- Verify reasoning is 150-400 words (not empty, not too short/long)
- Verify Judge reasoning includes actionable guidance for user decision-making
- Verify risk_label is defensible from Prosecutor/Defender scores and code evidence

## 9. NON-FUNCTIONAL REQUIREMENTS

### 9.1 Performance
- Single verdict (A05 with 5 findings) should complete in < 30 seconds
- Full pipeline (all 3 categories) should complete in < 2 minutes
- LLM calls should not timeout (use 60-second timeout per call)

### 9.2 Robustness
- Invalid JSON from LLM: parser should attempt recovery (JSON extraction from surrounding text)
- Network failures: implement retry logic with exponential backoff (2 retries max)
- Missing fields: use sensible defaults:
  - confidence_score: default to 50 if missing (uncertain)
  - reasoning: default to "Analysis incomplete" if missing
- Out-of-range confidence: clamp to 1-100

### 9.3 Logging & Debugging
- Log each step with confidence scores:
  - Prosecutor scored {score}/100: "{reasoning[:100]}..."
  - Defender scored {score}/100: "{reasoning[:100]}..."
  - Judge decided {score}/100: "{reasoning[:100]}..."
- Include LLM prompt and response (truncated if > 1000 chars)
- Log any parsing errors with full context

### 9.4 Code Quality
- All functions must have docstrings describing purpose, inputs, outputs
- All Pydantic models must have field descriptions
- All error cases must be handled (no bare exceptions)
- Type hints on all parameters and return values

## 12. SUCCESS CRITERIA

1. ✅ All three agents (Prosecutor, Defender, Judge) output valid JSON with:
   - confidence_score: int 1-100
   - reasoning: str 150-400 words
   
2. ✅ Prosecutor confidence aligns with evidence:
   - 85+ only for obvious exploits
   - 50-69 for uncertain findings
   - <30 for likely false positives

3. ✅ Defender confidence can disagree with Prosecutor:
   - Can score 90 while Prosecutor scored 80 (agreement)
   - Can score 20 while Prosecutor scored 85 (disagreement)
   - Reasoning explains the difference

4. ✅ Judge assigns risk labels (not auto-derived from scores):
   - critical_risk: Clear evidence, user should fix immediately
   - medium_risk: Has merit, user should review and decide
   - low_risk: Minor or contextual, user may defer
   - false_positive: Unlikely to be real risk, user can close
   - Reasoning explains why this label and what user should do

5. ✅ Risk labels are defensible and actionable:
   - Prosecutor 85+ + Defender <40 → critical_risk (clear action)
   - Prosecutor 70-84 or mixed → medium_risk (user decision needed)
   - Prosecutor <50 + Defender >60 → low_risk (user can defer)
   - Prosecutor <30 or clear defense → false_positive (user can close)

6. ✅ End-to-end pipeline runs without errors

7. ✅ Risk labels are deterministic (same diff = same risk label with T=0.0)

8. ✅ Judge reasoning is actionable:
   - Clearly states what user should do (fix, review, defer, close)
   - Explains which arguments were stronger
   - Cites specific code evidence or context

9. ✅ No hallucinations (all reasoning traceable to input findings/code)
